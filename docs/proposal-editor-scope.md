# Proposal Editor — Scope & Build Plan

**Status:** scoped, not started · **Owner:** Maurice · **Drafted:** 2026-06-29

## What we're building

A native, Google-Docs-style **proposal editor** inside FASS Flow. FASS
generates the entire response — every volume, every required section, an
automatic table of contents — and the user **reviews** rather than writes
from scratch. The portions that genuinely need a human (AI-drafted prose,
placeholders, low-confidence parses) are **highlighted for review**, and the
user can **comment, accept, or edit** those portions inline.

This is explicitly NOT a Google Docs integration. It's our own editor — the
differentiating, "next-level" feature. No Google Cloud, no OAuth.

The one-line pitch: *paste a solicitation → get a formatted, compliant draft
proposal you only have to review the highlights of.*

## What already exists (reuse, don't rebuild)

From `src/pages/Fill.jsx` + its libs — the pipeline is already half here:

- `src/lib/solicitationParser.js` → `parseSolicitation()` extracts **volumes**,
  due date, page limits, **format rules** (font, spacing, margins), required
  docs, and **eval criteria** (Section L/M). `buildOutline()` turns that into
  an ordered section/compliance list.
- `src/lib/aiClient.js` → `draftSection({ sectionName, sectionDescription })`
  already drafts a single section grounded in the user's past performance.
  `analyzeSolicitation()` and `extractFromImages()` exist too.
- Backend `app/routers/ai.py` already serves the draft/score endpoints.
- The compliance matrix, page-limit awareness, and capability statement are
  built. Confidence scoring (`parseConfidence`) is built.

**The gap:** all of that produces a *checklist + isolated section drafts*.
Nothing assembles it into one formatted document, generates a TOC, renders it
in an editor, flags review spans, or supports comments/export. That's this
project.

## Architecture

### Editor library — TipTap (ProseMirror)

Recommend **TipTap**. Rationale:
- ProseMirror under the hood = the same engine serious doc editors use;
  structured document model (not contenteditable soup), so headings, volumes,
  page-limit counting, and anchored comments are all tractable.
- First-class **marks** → exactly how we implement review-highlights and
  comment anchors (a mark carries an id that ties a text range to a comment
  thread, survives edits).
- Headings are real nodes → **auto-TOC** is a cheap traversal of the doc.
- Collaboration extension exists if we ever want realtime (we already run
  Supabase Realtime for the messenger — same pattern).
- React bindings are clean; bundle is reasonable.

Alternative considered: **Lexical** (Meta). Comparable, slightly lighter, but
weaker out-of-the-box comment/suggestion ecosystem. TipTap wins on the comment
layer, which is the heart of this feature.

`npm i @tiptap/react @tiptap/starter-kit @tiptap/extension-highlight` plus a
small custom comment-mark extension.

### Data model (Supabase)

```
proposal_documents
  id, user_id, proposal_id (FK -> existing proposals), title,
  format jsonb,            -- font/spacing/margins/page-limits from parse
  created_at, updated_at

proposal_sections
  id, document_id, volume,         -- "Volume I — Technical Approach"
  heading, order_index,
  body jsonb,                      -- ProseMirror/TipTap JSON
  source text,                     -- 'ai' | 'deterministic' | 'user' | 'placeholder'
  review_status text,              -- 'needs_review' | 'approved'
  page_estimate int

proposal_comments
  id, document_id, section_id, anchor jsonb,  -- mark id + text excerpt
  author_id, body, status,         -- 'open' | 'resolved'
  parent_id,                       -- threading/replies
  created_at
```

Reuse the existing `proposals` row as the parent so a document stays tied to
the opportunity it came from (Pipeline, WARDOG, R-E-A-D all already key off
`proposals`).

### Generation flow (the magic)

1. User has a parsed solicitation (today's Fill.jsx output).
2. **Assemble**: for each volume in `parsed.volumes`, create sections from
   `buildOutline()`; for each draftable section call the existing
   `draftSection()`; deterministic sections (cover page, cert page, TOC) are
   templated, not AI'd.
3. **Mark review spans**: anything `source === 'ai'`, any placeholder
   (`[INSERT past performance reference]`), and any section derived from a
   low-confidence parse field gets `review_status = 'needs_review'` and a
   highlight mark. Everything else renders clean.
4. **TOC**: generated by walking heading nodes → rendered at the top, kept in
   sync as sections are added/removed.
5. **Page-limit guardrails**: per-volume running page estimate vs the parsed
   limit; warn in the gutter when a volume is over (we already parse the
   limits, just need to surface them live).

### Review / comment UX

- Left: the document (volumes, TOC, formatted body).
- Right: a comment rail (the layout from the approach mockup).
- Highlighted spans = needs-review. Clicking one opens its thread; user can
  **Approve** (clears the highlight), **Comment** (anchored thread, replies),
  or **Edit** inline (TipTap editing, marks the section `source: 'user'`).
- A top progress chip: "12 of 18 sections reviewed" — mirrors the
  bid-checklist pattern already in Pipeline, so it feels native.

### Export (compliance is the moat)

- Assemble the approved document to **.docx** honoring the parsed format rules
  (font, spacing, margins) and respecting per-volume page limits — this is
  where govcon proposals actually get rejected, so enforcing it is the
  differentiator.
- Backend generation (python-docx) keeps fonts/margins exact; the `docx` skill
  pattern already in this environment is the reference.
- PDF as a secondary export.

## Phased build

- **Phase 1 — Assemble & render.** TipTap editor, volumes + auto-TOC, generate
  all sections from the existing parse/draft pipeline, highlight needs-review
  spans. Read + inline edit. (This alone is demoable and a huge leap.)
- **Phase 2 — Comments & review loop.** Anchored comment threads, approve/
  resolve, the review-progress chip, per-volume page-limit warnings.
- **Phase 3 — Compliant export.** .docx with exact format rules + page-limit
  enforcement; PDF.
- **Phase 4 — (optional) Realtime co-review.** Reuse Supabase Realtime so a
  teammate/BD Partner can comment live.

## Open product decisions for next session

1. **Entry point:** does this replace the FASS FILL "Compliance & Outline" tab,
   or become a new "Draft" tab/step after the matrix? (Lean: a new step — keep
   the matrix as the pre-flight, the editor as the build.)
2. **Who can comment?** Just the owner first, or also a BD Partner / Team Up
   collaborator (we have the relationships already).
3. **AI re-draft from a comment:** should "reject + comment" auto-trigger a
   re-draft of that section from the note? (High-wow, Phase 2.5.)

## Effort estimate (rough)

- Phase 1: ~1 focused session (editor + assemble + TOC + highlights).
- Phase 2: ~1 session (comments are the bulk).
- Phase 3: ~half session (export, reusing docx tooling).

No external dependencies, no credentials to provision — it's all our stack
(React + TipTap, FastAPI, Supabase), extending code that already exists.
